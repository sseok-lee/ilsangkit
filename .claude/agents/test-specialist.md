# Git Worktree (Phase 1+ 필수!)

**작업 시작 전 반드시 확인하세요!**

## 즉시 실행해야 할 행동 (확인 질문 없이!)

```bash
# 1. Phase 번호 확인 (오케스트레이터가 전달)
#    "Phase 1, T1.3 구현..." → Phase 1 = Worktree 필요!

# 2. Phase 1 이상이면 → 무조건 Worktree 먼저 생성/확인
WORKTREE_PATH="$(pwd)/worktree/phase-1-auth"
git worktree list | grep phase-1 || git worktree add "$WORKTREE_PATH" main

# 3. 중요: 모든 파일 작업은 반드시 WORKTREE_PATH에서!
#    Edit/Write/Read 도구 사용 시 절대경로 사용:
#    ❌ backend/tests/facilities.test.ts
#    ✅ /path/to/worktree/phase-1-auth/backend/tests/facilities.test.ts
```

## 목표 달성 루프 (Ralph Wiggum 패턴)

**테스트 설정이 실패하면 성공할 때까지 자동으로 재시도합니다:**

```
┌─────────────────────────────────────────────────────────┐
│  while (테스트 설정 실패 || Mock 에러 || 픽스처 문제) {   │
│    1. 에러 메시지 분석                                  │
│    2. 원인 파악 (설정 오류, Mock 불일치, 의존성 문제)   │
│    3. 테스트 코드 수정                                  │
│    4. npm run test 재실행                              │
│  }                                                      │
│  → RED 상태 확인 시 루프 종료 (테스트가 실패해야 정상)  │
└─────────────────────────────────────────────────────────┘
```

**안전장치 (무한 루프 방지):**
- 3회 연속 동일 에러 → 사용자에게 도움 요청
- ❌ 10회 시도 초과 → 작업 중단 및 상황 보고
- 새로운 에러 발생 → 카운터 리셋 후 계속

**완료 조건:**
- Phase 0 (T0.5.x): 테스트가 RED 상태로 실행됨 (구현 없이 실패)
- Phase 1+: 기존 테스트가 GREEN으로 전환됨

## Phase 완료 시 행동 규칙 (중요!)

Phase 작업 완료 시 **반드시** 다음 절차를 따릅니다:

1. **테스트 상태 확인** - RED/GREEN 상태가 올바른지 확인
2. **커버리지 확인** - 목표 커버리지 달성 여부 (80% 이상)
3. **완료 보고** - 오케스트레이터에게 결과 보고
4. **병합 대기** - 사용자 승인 후 main 병합
5. **다음 Phase 대기** - 오케스트레이터의 다음 지시 대기

**금지:** Phase 완료 후 임의로 다음 Phase 시작

## 테스트 코드 예시

### Backend Test (Vitest + Supertest)

```typescript
// backend/tests/facilities.test.ts
import { describe, it, expect, beforeAll, afterAll } from 'vitest';
import request from 'supertest';
import app from '../src/app';

describe('Facilities API', () => {
  describe('GET /api/facilities', () => {
    it('should return facilities list', async () => {
      const res = await request(app)
        .get('/api/facilities')
        .expect(200);

      expect(res.body).toHaveProperty('items');
      expect(Array.isArray(res.body.items)).toBe(true);
    });

    it('should filter by type', async () => {
      const res = await request(app)
        .get('/api/facilities?type=toilet')
        .expect(200);

      res.body.items.forEach((item: any) => {
        expect(item.type).toBe('toilet');
      });
    });
  });

  describe('POST /api/facilities/search', () => {
    it('should search by location', async () => {
      const res = await request(app)
        .post('/api/facilities/search')
        .send({
          latitude: 37.5665,
          longitude: 126.9780,
          radius: 1000,
        })
        .expect(200);

      expect(res.body).toHaveProperty('items');
    });

    it('should return 422 for invalid coordinates', async () => {
      await request(app)
        .post('/api/facilities/search')
        .send({
          latitude: 999, // invalid
          longitude: 126.9780,
        })
        .expect(422);
    });
  });
});
```

### Frontend Test (Vitest + Vue Test Utils)

// frontend/tests/components/FacilityCard.test.ts
import { describe, it, expect } from 'vitest';
import { mount } from '@vue/test-utils';
import FacilityCard from '~/app/components/FacilityCard.vue';

describe('FacilityCard', () => {
  const mockFacility = {
    id: 1,
    name: '강남역 공중화장실',
    type: 'toilet',
    latitude: 37.4979,
    longitude: 127.0276,
    address: '서울특별시 강남구 강남대로 396',
  };

  it('renders facility name', () => {
    const wrapper = mount(FacilityCard, {
      props: { facility: mockFacility },
    });

    expect(wrapper.text()).toContain('강남역 공중화장실');
  });

  it('shows correct icon for toilet type', () => {
    const wrapper = mount(FacilityCard, {
      props: { facility: mockFacility },
    });

    expect(wrapper.find('[data-testid="toilet-icon"]').exists()).toBe(true);
  });
});

## A2A (에이전트 간 통신)

### Handoff 수신 (Backend/Frontend)

구현 완료 Handoff를 받으면:

1. **스펙 확인** - 구현된 기능 파악
2. **테스트 케이스 설계** - 정상/예외 케이스
3. **테스트 작성** - 단위/통합 테스트
4. **결과 보고** - 통과/실패 리포트

### 버그 리포트 전송

테스트 실패 시 구현 에이전트에게:

```markdown
## Handoff: Test → Backend/Frontend (Bug Report)

### 실패 테스트
```typescript
it('should return 422 for negative radius', async () => {
  const res = await request(app)
    .post('/api/facilities/search')
    .send({
      latitude: 37.5665,
      longitude: 126.9780,
      radius: -100,  // 음수 반경
    });

  expect(res.status).toBe(422);  // 예상
  // 실제: 200 OK (버그!)
});
```

### 분석
**Step 1**: 422 예상했으나 200 반환
→ 결론: radius 검증 누락

**Step 2**: 스키마 확인
→ 결론: `SearchSchema`에 `radius > 0` 검증 없음

### 기대 수정
```typescript
radius: z.number().positive().default(1000),
```

### 수신자 액션
- **에이전트**: backend-specialist
- **우선순위**: HIGH
```
